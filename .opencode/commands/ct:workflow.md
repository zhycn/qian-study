---

编排完整的词条处理工作流，串联多个命令自动执行。

如果用户未指定范围，询问：

1. 工作流范围？
   - `empty`：所有空壳词条（正文 < 300 字）
   - `category:xxx`：指定分类的所有词条
   - `all`：全部词条
   - 手动指定列表
2. 评分阈值？（默认 3.0/5）
3. 目标深度？（默认 beginner）
4. 是否自动修复？（默认 true，不合格自动触发 deep-write）

## 工作流流程

```
扫描词条 → 逐个审查(ct:audit) → 评分判定 → 不合格触发 deep-write → 二次审查 → 交叉引用(ct:link) → 验证构建
```

## 执行步骤

### 步骤 1：确定目标列表

- `empty`：用 Glob 扫描 `docs/**/*.md`，筛选正文字数 < 300 的
- `category:xxx`：从 `docs/glossary/index.md` 提取对应分类词条
- `all`：扫描 `docs/basics/`、`docs/money-banking/`、`docs/investing/` 等所有分类目录
- 手动列表：直接使用

输出目标清单供确认。

### 步骤 2：逐词条执行工作流

对每个词条执行：

1. **审查**：调用 `/ct:audit <词条路径> --threshold <阈值>`
2. **判定**：
   - 通过 → 标记 ✅，进入下一步
   - 不合格 → 标记 ⚠️，进入步骤 3
3. **重写**（仅不合格词条）：调用 `/ct:deep-write <词条路径> --depth <深度>`
4. **二次审查**：重写后再次调用 `/ct:audit`
   - 通过 → 标记 ✅
   - 仍不合格 → 标记 ❌（需人工介入）
5. **交叉引用**：调用 `/ct:link --add <词条名>` 补充关联

### 步骤 3：生成报告

```markdown
## 工作流完成报告

### 处理概况

- 目标词条：X 个
- 直接通过：X 个
- 重写后通过：X 个
- 需人工介入：X 个
- 失败：X 个

### 逐词条结果

| 词条  | 初次评分 | 判定   | 二次评分 | 最终状态     |
| ----- | -------- | ------ | -------- | ------------ |
| 词条A | 4.2/5    | 通过   | -        | ✅           |
| 词条B | 2.1/5    | 不合格 | 4.0/5    | ✅（重写）   |
| 词条C | 1.8/5    | 不合格 | 2.5/5    | ❌（需人工） |

### 验证

- lint:check ✅/❌
- type-check ✅/❌
- docs:build ✅/❌
```

### 步骤 4：验证

按 AGENTS.md 验证顺序运行：`pnpm lint:check` → `pnpm type-check` → `pnpm docs:build`。

### 步骤 5：后续操作

询问用户是否需要：

- 通过 `/vp:commit-push` 提交变更
- 对标记 ❌ 的词条进行人工处理
- 重新运行工作流（调整阈值或深度）

## 工具绑定

| 步骤     | 工具                |
| -------- | ------------------- |
| 扫描词条 | Glob + Read         |
| 审查评分 | 调用 /ct:audit      |
| 深度写作 | 调用 /ct:deep-write |
| 交叉引用 | 调用 /ct:link       |
| 验证构建 | Bash                |

## 容错

- 单词条处理失败 → 标记失败，继续处理下一个
- deep-write 失败 → 标记 ❌，不阻塞其他词条
- `docs:build` 失败 → 输出错误日志，建议逐个排查
- 工作流中断可恢复，已完成的词条不会丢失
- 二次审查仍不合格 → 标记"需人工介入"，不无限循环

## 示例

- `/ct:workflow empty` — 处理所有空壳词条
- `/ct:workflow category:基础概念 --threshold 3.5 --depth intermediate` — 指定分类、阈值、深度
- `/ct:workflow all --no-auto-fix` — 全部词条，只审查不自动重写
- `/ct:workflow 期权,期货,互换` — 手动指定词条列表
