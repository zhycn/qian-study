# 开放银行

> 银行把自己的能力"拆开"，让第三方也能调用

## 概述

开放银行（Open Banking）是指银行通过 API（应用程序编程接口）将账户信息、支付能力、信贷服务等开放给第三方开发者，让非银行机构也能为用户提供金融服务。

就像手机有 App Store，开放银行让各种金融"小程序"可以接入银行的核心能力。用户在第三方 App 中就能查看多家银行账户、完成转账、申请贷款，无需登录每家银行的 App。

## 为什么重要

开放银行正在**打破金融服务的边界**：

- **用户体验**：在一个 App 里管理多家银行的账户，告别"多 App 切换"
- **创新加速**：第三方开发者可以基于银行 API 创造新产品，降低金融创新门槛
- **竞争促进**：倒逼传统银行提升服务质量，打破"大而不能倒"的垄断格局
- **数据价值**：在用户授权下实现数据的安全流转，释放金融数据的生产力
- **普惠金融**：让中小微企业更容易获得信贷，第三方平台利用多银行数据做出更准确的信用评估

## 核心原理

开放银行的技术架构基于 API 经济：

**API 网关**（API Gateway）：银行对外提供标准化 API 接口，包括账户查询、支付发起、转账执行、信贷申请等。API 网关负责身份认证、权限控制、流量管理和日志记录。

**OAuth 2.0 授权**：用户通过 OAuth 协议授权第三方应用访问其银行数据。授权过程包括：

1. 用户在银行页面登录并确认授权范围
2. 银行颁发 Access Token 给第三方应用
3. 第三方应用使用 Token 调用银行 API
4. Token 有过期时间，用户可随时撤销授权

**数据标准化**：不同银行的 API 格式不同，需要统一标准。国际上有 Open Banking Standard（英国）、Berlin Group（欧洲）等标准组织。

**安全机制**：

- **强客户认证**（SCA，Strong Customer Authentication）：欧盟 PSD2 要求双因素认证
- **数据加密**：传输层 TLS 加密，存储层数据脱敏
- **审计追踪**：所有 API 调用记录可追溯，满足合规要求

## 技术/类型详解

开放银行 API 按功能分类：

**账户信息 API**：

- 查询账户余额、交易明细、账单
- 典型应用：个人财务管理 App（如 Mint、随手记）

**支付发起 API**：

- 直接从银行账户发起支付，绕过信用卡网络
- 典型应用：电商结账、账单支付

**信贷产品 API**：

- 第三方平台基于用户银行数据推荐贷款产品
- 典型应用：贷款比价平台、嵌入式金融（Embedded Finance）

**身份验证 API**：

- 银行作为身份提供商，验证用户身份
- 典型应用：KYC 流程简化、反欺诈

| 维度     | 欧盟（PSD2） | 英国（Open Banking） | 中国         | 美国         |

| 驱动方式 | 法规强制     | 行业自律+监管        | 市场驱动     | 市场驱动     |

| 成熟度   | 高           | 高                   | 中           | 中           |

## 实战应用

- **账户聚合**：用户在单个 App 中查看所有银行账户余额和交易
- **智能记账**：自动分类消费记录，提供预算管理和财务建议
- **场景金融**：电商、出行、外卖等场景中嵌入支付和信贷服务
- **中小企业贷款**：基于企业银行流水数据，快速审批贷款

## 关键要点

- 核心技术是 API，需要严格的安全和权限管理
- 欧盟 PSD2 法规是开放银行的重要推动力
- 用户授权是开放银行的前提，数据所有权归用户
- 数据隐私和安全是最大挑战

## 常见误区

- **误区一**：开放银行等于数据泄露——开放银行基于用户授权和严格的安全标准，数据流转全程加密可追溯
- **误区二**：开放银行只利好科技公司——传统银行也能通过开放 API 拓展业务场景，增加收入来源
- **误区三**：开放银行在中国没有发展——中国也在积极探索，银行业协会发布《开放银行 API 安全规范》，监管框架逐步完善
- **误区四**：开放银行就是 API 开放——开放银行不仅是技术开放，更是商业模式和组织文化的变革
